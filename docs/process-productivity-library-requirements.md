# 工艺工效库页面需求文档

版本：v1.0

日期：2026-06-11

适用范围：桥梁下部结构自动排程系统实际工程化版本

## 1. 页面定位

工艺工效库页面是排程系统的工艺模板与工效参数维护页面，负责维护“构件类型 -> 施工工艺 -> 工效分组 -> 默认资源”的配置关系。项目参数页面选择构件工艺和工效时，必须引用本页面维护的工艺工效库；后续生成任务、计算工期、匹配资源时，也以该库为规则来源。

当前 demo 中，该页已从历史的“结构类型、施工工艺、工效”三列表结构升级为工程化模型：施工工艺作为一级模板，现有工效作为该工艺的“默认工效”，并支持在同一施工工艺下新增多个工效分组。实际工程中，后端已存在历史数据结构，应在加载时迁移/映射为新的工艺工效库结构，并保持历史数据可追溯。

## 2. 页面主要功能

### 2.1 工艺工效库展示

页面以表格展示施工工艺及工效库，核心列包括：

| 列 | 含义 | 规则 |
| --- | --- | --- |
| 构件 | 工艺适用的结构构件类型 | 如桩基、承台、墩柱、盖梁、桥台 |
| 工艺名称 | 具体施工工艺 | 如旋挖钻成孔、冲击钻成孔、人工挖孔 |
| 工期算法 | 当前默认工效的计算方式 | 按日完成量、按单位耗时、固定工期 |
| 工程量来源 | 当前默认工效使用的工程量字段 | 桩长、根数、墩高、构件数量 |
| 工效分组 | 该施工工艺下的多个工效参数 | 包含分组名、工效值、单位、默认标记 |
| 默认资源 | 该工艺默认占用的资源类型 | 与资源配置页面的资源池关联 |

页面应支持纵向滚动，适配历史数据中较多结构类型和施工工艺的情况。

### 2.2 施工工艺维护

用户可维护施工工艺名称和默认资源类型。

操作规则：

- 工艺名称可编辑，修改后影响项目参数页展示和后续任务 `process_name`。
- 默认资源从资源池中选择，修改后影响任务候选资源类型。
- 构件类型原则上不在页面内随意修改，避免破坏历史数据和构件匹配关系。
- 工艺 ID、构件类型、方法 ID 应作为稳定标识由后端维护。

### 2.3 工效分组维护

每个施工工艺下支持一个或多个工效分组。

用户操作包括：

- 修改工效分组名称。
- 修改工效数值。
- 修改桩基工效单位。
- 新增工效分组。
- 删除非唯一工效分组。
- 将某个分组设为默认工效。
- 保存工艺工效库到后端。

新增分组规则：

1. 新增分组默认复制当前默认工效的工期算法、工程量来源、工效值和单位。
2. 新增分组名称默认生成，如“工效分组2”。
3. 新增分组默认不是默认工效，用户可手动设为默认。
4. 每个施工工艺下必须至少保留一个工效分组。

默认工效规则：

1. 每个施工工艺下必须且只能有一个默认工效。
2. 历史数据升级后，原有工效自动成为该施工工艺的默认工效。
3. 当默认工效变化时，施工工艺模板上的 `duration_method`、`quantity_source`、`productivity_value`、`productivity_unit` 应同步为默认工效的值。
4. 项目参数页未指定具体工效分组时，使用默认工效。

### 2.4 保存与同步

页面提供“保存到后端/保存到 Supabase”操作。

交互规则：

- 进入页面时，若后端已有工艺工效库数据，优先加载后端数据。
- 若后端没有数据，可使用系统内置默认工艺工效库初始化。
- 用户修改工艺名称、资源类型、工效分组后，页面标记为未同步。
- 保存成功后，页面标记为已同步。
- 保存失败时保留前端修改，并展示错误信息。

## 3. 本次变化功能

### 3.1 桩基增加人工挖孔施工工艺

桩基构件类型在既有旋挖钻、回旋钻/冲击钻等历史工艺基础上，新增“人工挖孔”施工工艺。

建议工艺模板：

| 字段 | 建议值 |
| --- | --- |
| 构件类型 | 桩基 |
| 工艺名称 | 人工挖孔 |
| 方法 ID | `manual_pile` |
| 默认工期算法 | 按单位耗时计算 |
| 默认工程量来源 | 桩长 |
| 默认工效单位 | 天/m |
| 默认资源类型 | 人工挖孔班 |

规则：

- 项目参数页应能将桩基构件工艺切换为人工挖孔。
- AI 快捷设置应能识别“人工挖孔”“挖孔桩”等描述并匹配该工艺。
- 选择人工挖孔后，生成任务时应使用人工挖孔对应的资源类型。

### 3.2 桩基工效单位增加 m/天

历史数据中，桩基常见工效单位为“天/根”，含义是每根桩需要多少天。新版本在此基础上增加“m/天”，含义是每天完成多少米桩长。

桩基核心单位及计算逻辑：

| 单位 | 工期算法 | 工程量来源 | 工期计算 |
| --- | --- | --- | --- |
| 天/根 | 按单位耗时计算 | 根数 | `ceil(根数 * 工效值)` |
| m/天 | 按日完成量计算 | 桩长 | `ceil(桩长 / 工效值)` |

示例：

- 某桩基按“天/根”计算，工效为 2 天/根，工程量为 1 根，则工期为 2 天。
- 某桩基按“m/天”计算，工效为 20 m/天，桩长为 38 m，则工期为 2 天。

规则：

- 桩基工效单位切换为“天/根”时，系统自动将 `duration_method` 设置为 `days_per_unit`，`quantity_source` 设置为 `count`。
- 桩基工效单位切换为“m/天”时，系统自动将 `duration_method` 设置为 `units_per_day`，`quantity_source` 设置为 `pile_length_m`。
- 不同单位的工效值不能直接比较大小，应按对应计算逻辑生成工期。
- 历史“天/根”数据迁移后仍保持原计算语义。

扩展说明：系统内部可继续兼容“根/天”“天/m”等单位，但本期页面验收重点为“天/根”和“m/天”两类计算逻辑。

### 3.3 现有施工工艺升级为默认工效

历史结构如截图所示，通常为：

| 结构类型 | 施工工艺 | 工效 |
| --- | --- | --- |
| 桩基 | 旋挖钻 | 3 天/根 |
| 桩基 | 回旋钻 | 2 天/根 |
| 桩基 | 冲击钻 | 2 天/根 |
| 墩身 | 爬模施工 | 7 天/节 |
| 桥台 | -- | 15 天/个 |

升级后，每一条历史施工工艺应映射为一个 `ProcessTemplate`，原工效映射为该工艺下的一个 `ProductivityOption`，分组名为“默认工效”。

映射规则：

| 历史字段 | 新模型字段 |
| --- | --- |
| 结构类型 | `component_type` |
| 施工工艺 | `process_name` |
| 工效数值 | `productivity_options[].productivity_value` |
| 工效单位 | `productivity_options[].productivity_unit` |
| 工效计算方式 | `productivity_options[].duration_method` |
| 工程量来源 | `productivity_options[].quantity_source` |
| 是否默认 | `productivity_options[].is_default = true` |

升级后，用户可在同一施工工艺下新增更多工效分组，例如“默认工效”“软岩地层”“硬岩地层”“雨季折减”等。

## 4. 数据来源与规则

### 4.1 后端数据来源

实际工程后端已有历史工效数据结构，页面应通过后端接口获取标准化后的工艺工效库。后端负责完成历史结构到新结构的映射。

当前 demo 中，工艺工效库支持以下来源优先级：

1. 数据库工艺关系表，例如 `public.process_relationships`。
2. 数据库无数据时，使用系统默认工艺工效库初始化。
3. 页面保存后，将标准化工艺工效库写回后端。

### 4.2 标准数据模型

施工工艺模板 `ProcessTemplate`：

| 字段 | 含义 |
| --- | --- |
| `id` | 工艺模板稳定 ID |
| `component_type` | 构件类型 |
| `process_name` | 施工工艺名称 |
| `method_id` | 工艺方法 ID，可用于项目参数页匹配 |
| `duration_method` | 默认工期算法 |
| `quantity_source` | 默认工程量来源 |
| `productivity_value` | 默认工效值 |
| `productivity_unit` | 默认工效单位 |
| `resource_type` | 默认资源类型 |
| `productivity_options` | 工效分组列表 |
| `applicability` | 适用条件扩展字段 |
| `is_default` | 是否为该构件类型默认工艺 |

工效分组 `ProductivityOption`：

| 字段 | 含义 |
| --- | --- |
| `id` | 工效分组稳定 ID |
| `name` | 工效分组名称 |
| `duration_method` | 工期算法 |
| `quantity_source` | 工程量来源 |
| `productivity_value` | 工效值 |
| `productivity_unit` | 工效单位 |
| `is_default` | 是否为该施工工艺默认工效 |

### 4.3 工期算法规则

支持三类工期算法：

| 算法 | 说明 | 公式 |
| --- | --- | --- |
| `units_per_day` | 按日完成量计算 | `ceil(工程量 / 工效值)` |
| `days_per_unit` | 按单位耗时计算 | `ceil(工程量 * 工效值)` |
| `fixed_days` | 固定工期 | `ceil(工效值)` |

计算结果最小为 1 天。

### 4.4 工程量来源规则

| 来源 | 含义 | 适用构件 |
| --- | --- | --- |
| `count` | 构件数量 | 桩基按根、承台/桥台/盖梁等按个 |
| `pile_length_m` | 桩长 | 桩基 |
| `pier_height_m` | 墩高 | 墩柱 |

桩基按“天/根”计算时，工程量为 1 根；按“m/天”计算时，工程量为单根桩长。

### 4.5 默认工艺与默认工效规则

- 一个构件类型可以有多个施工工艺。
- 一个构件类型建议有一个默认施工工艺。
- 一个施工工艺可以有多个工效分组。
- 一个施工工艺必须有且只有一个默认工效分组。
- 当项目参数页的构件未指定工艺时，使用构件类型默认工艺。
- 当项目参数页的构件指定工艺但未指定工效分组时，使用该工艺默认工效。

## 5. 接口要求

### 5.1 获取工艺工效库

```http
GET /api/process-library
```

响应：

```json
[
  {
    "id": "pile_rotary_regular",
    "component_type": "pile",
    "process_name": "旋挖钻成孔",
    "method_id": "rotary_drill",
    "duration_method": "units_per_day",
    "quantity_source": "pile_length_m",
    "productivity_value": 18,
    "productivity_unit": "m/天",
    "resource_type": "rotary_drill",
    "productivity_options": [
      {
        "id": "pile_rotary_regular-default",
        "name": "默认工效",
        "duration_method": "units_per_day",
        "quantity_source": "pile_length_m",
        "productivity_value": 18,
        "productivity_unit": "m/天",
        "is_default": true
      }
    ],
    "applicability": {},
    "is_default": true
  }
]
```

### 5.2 保存工艺工效库

```http
PUT /api/process-library
```

请求：

```json
{
  "process_library": []
}
```

保存规则：

- 后端应校验 `process_library` 非空。
- 后端应保证每个工艺至少有一个工效分组。
- 后端应保证每个工艺只有一个默认工效分组。
- 后端保存时同步工艺模板默认字段和默认工效字段。
- 后端不应向浏览器暴露数据库写入凭据。

### 5.3 项目参数页引用规则

项目参数页通过当前 `ScenarioInput.process_library` 引用工艺工效库。

引用规则：

- 工艺下拉选项按构件类型过滤。
- 工效下拉选项按当前已选工艺过滤。
- 工艺变化时，工效默认切换为新工艺的默认工效。
- 工效变化时，仅更新当前构件的 `productivity_option_id`。

## 6. 页面交互与异常处理

### 6.1 表单校验

- 工艺名称不能为空。
- 工效分组名称不能为空，空值可自动显示为“工效”但保存时建议补齐。
- 工效值必须大于 0。
- 每个工艺至少保留一个工效分组。
- 删除默认工效时，系统应自动将剩余第一个分组设为默认。
- 保存前若存在不合法配置，应阻止保存并提示用户。

### 6.2 状态提示

- 未修改时显示“已同步”。
- 修改后显示“保存到后端/保存到 Supabase”。
- 保存中显示加载状态。
- 保存失败展示后端错误。

### 6.3 对排程结果的影响

修改工艺工效库后，应清空已生成的任务图、求解结果和方案对比结果。原因是：

- 工艺名称变化会影响任务展示。
- 资源类型变化会影响资源约束。
- 默认工效变化会影响未指定工效分组构件的工期。
- 工效单位变化会影响工期计算逻辑。

## 7. 验收标准

功能验收：

- 能展示后端历史数据升级后的施工工艺和默认工效。
- 桩基工艺中能看到并选择“人工挖孔”。
- 桩基工效单位支持“天/根”和“m/天”，并能自动切换对应工期算法和工程量来源。
- 现有施工工艺的历史工效自动成为“默认工效”。
- 用户可在同一施工工艺下新增、删除、编辑工效分组。
- 用户可将非默认工效分组设为默认。
- 页面能保存工艺工效库，并在刷新后保持数据。

计算验收：

- 桩基使用“天/根”时，工期按 `ceil(根数 * 工效值)` 计算。
- 桩基使用“m/天”时，工期按 `ceil(桩长 / 工效值)` 计算。
- 固定工期类构件按 `ceil(工效值)` 计算。
- 工期计算结果最小为 1 天。

联动验收：

- 项目参数页切换构件工艺后，可看到该工艺下的工效分组。
- 项目参数页未指定工效分组时，使用该工艺默认工效。
- 修改默认工效后，未指定工效分组的构件在重新生成任务时使用新默认值。
- 修改资源类型后，重新生成任务使用新的候选资源类型。

数据验收：

- 历史“结构类型-施工工艺-工效”数据能无损映射为新模型。
- `productivity_options` 可持久化多个分组。
- 每个工艺保存后仍保持一个且仅一个默认工效。
- 后端返回数据可被前端直接渲染，无需前端理解历史表结构。

## 8. 默认假设

- 后端负责历史数据结构升级和标准化输出。
- 前端不直接读写历史表，只读写标准化后的工艺工效库接口。
- 本期重点支持桥梁下部结构构件，后续可扩展路基、隧道、上部结构工艺。
- 工艺工效库是全局或项目级配置，具体作用域由后端权限和数据隔离策略决定。
