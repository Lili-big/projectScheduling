# 项目参数页面需求文档

版本：v1.0

日期：2026-06-11

适用范围：桥梁下部结构自动排程系统实际工程化版本

## 1. 页面定位

项目参数页面是排程系统的输入总览与首轮配置页面，负责把实际工程已有的结构物参数转换为可排程的结构树，并让用户在结构清单上完成工艺、工效的核对与局部调整。

当前 demo 中，该页承接“结构物参数 -> 工艺工效库 -> 工艺逻辑 -> 资源配置 -> 排程求解”的主链路。实际工程中，结构物参数不再以 Excel 为主来源，而应通过项目结构物参数 API 获取。页面侧重点从“导入结构物参数”调整为“展示权威结构物参数、校核排程输入、补充工艺工效选择”。

页面不负责完整录入结构物几何参数，也不替代工程结构物主数据系统。结构物几何参数以项目结构物参数服务为准；本页面仅维护排程需要的方案信息、工艺选择、工效分组和局部模拟参数。

## 2. 用户与使用场景

主要用户包括计划工程师、工区技术负责人、项目管理人员和排程系统管理员。

典型使用场景：

- 计划工程师进入项目后，检查系统从结构物参数 API 获取到的桥梁、幅别、墩台和构件清单是否完整。
- 技术负责人按施工方案调整部分构件工艺，例如将指定墩位桩基从默认旋挖钻调整为人工挖孔。
- 计划工程师选择不同工效分组，生成不同排程方案并对比工期、资源需求和里程碑偏差。
- 管理人员查看导入来源、结构物数量、跨径表达式和数据告警，判断当前参数是否可以进入求解。

## 3. 页面主要功能

### 3.1 基础参数维护

页面顶部展示并允许维护以下排程方案级参数：

| 字段 | 说明 | 规则 |
| --- | --- | --- |
| 方案名称 | 当前排程模拟方案名称 | 可编辑，修改后只影响当前方案 |
| 项目名称 | 当前项目名称 | 默认来自项目结构物参数 API |
| 计划开始 | 排程计划开始日期 | 后续任务日期以该日期为基准 |
| 求解时限 | 求解器单次运行时限，单位秒 | 必须大于 0 |

修改基础参数后，页面应清空已生成任务图、求解结果和方案对比结果，提示用户需要重新生成或求解。

### 3.2 结构构件清单展示

页面以表格展示结构物参数明细，列包括：

| 列 | 含义 | 来源 |
| --- | --- | --- |
| 工点 | 路、桥、隧等工点类型 | `ProjectBridge.workpoint_type` |
| 桥梁 / 工区 | 桥梁名称和工区/幅别名称 | `ProjectBridge.name`、`WorkSection.name` |
| 结构层级 | 下部结构或上部结构 | 下部来自 `Structure.components`，上部来自 `WorkSection.upper_structures` |
| 幅别 | 左幅、右幅、无幅别 | `WorkSection.side` |
| 位置 | 墩台号、跨号或结构位置 | `Structure.support_no`、`UpperStructureComponent.support_range` |
| 构件 | 构件实例名称 | `ComponentModel.name` |
| 类型 | 构件类型中文名 | `ComponentModel.component_type` 映射 |
| 结构尺寸 | 规范化尺寸摘要 | `ComponentModel.properties` |
| 工艺 | 当前构件选用工艺 | 由构件 `method_id` 与工艺库匹配 |
| 工效 | 当前构件选用工效分组 | 由构件 `productivity_option_id` 与工艺工效库匹配 |

下部结构构件是排程任务生成来源；上部结构当前仅展示，并用于辅助识别连续梁主墩等规则，不直接生成下部结构施工任务。

### 3.3 结构清单筛选

表格应支持按每一列进行筛选：

- 文本筛选：桥梁/工区、位置、构件、类型、结构尺寸、工艺、工效。
- 枚举筛选：结构层级、幅别。
- 筛选结果实时更新，并展示“当前筛选项数 / 总项数”。

筛选只影响页面展示，不改变底层结构物参数和排程输入。

### 3.4 工艺选择

当某个构件类型存在多个可选工艺时，用户可在结构清单中切换工艺。

当前默认支持：

| 构件类型 | 可选工艺示例 |
| --- | --- |
| 桩基 | 旋挖钻成孔、冲击钻成孔、人工挖孔 |
| 墩柱 | 墩柱施工、爬模施工等扩展工艺 |
| 承台、扩大基础、系梁、盖梁、桥台 | 默认施工工艺，可由工艺库扩展 |

工艺匹配规则：

1. 若构件存在 `method_id`，优先匹配工艺库中 `id` 或 `method_id` 相同的工艺。
2. 若构件没有 `method_id`，使用该构件类型下 `is_default = true` 的工艺。
3. 若没有默认工艺，使用该构件类型下第一个可用工艺。
4. 若没有任何匹配工艺，生成排程输入时应输出严重诊断，不允许静默求解。

切换工艺后，系统应同步选择该工艺的默认工效分组，并清空旧求解结果。

### 3.5 工效选择

当某个工艺存在多个工效分组时，用户可在结构清单中选择具体工效分组。工效分组决定工程量口径、工期算法和单位工效值。

桩基工效单位与计算规则：

| 工效单位 | 工期算法 | 工程量来源 |
| --- | --- | --- |
| m/天 | 按日完成量计算 | 桩长 |
| 根/天 | 按日完成量计算 | 根数 |
| 天/根 | 按单位耗时计算 | 根数 |
| 天/m | 按单位耗时计算 | 桩长 |

其他构件默认规则：

- 墩柱：优先按墩高计算。
- 承台、扩大基础、地系梁、中系梁、盖梁、桥台：通常按构件数量或固定工期计算。

工效匹配规则：

1. 若构件存在 `productivity_option_id`，优先匹配当前工艺下同 ID 的工效分组。
2. 若未指定或 ID 不存在，使用当前工艺下 `is_default = true` 的工效分组。
3. 若没有默认分组，使用当前工艺下第一个工效分组。

### 3.6 AI 快捷设置

页面提供自然语言快捷设置能力，用于批量修改构件工艺。

示例指令：

- “桩基默认采用旋挖钻施工，其中 1#墩-1 桩基、1#墩-2 桩基采用人工挖孔桩。”
- “左幅的 3#墩和 4#墩的桩基工艺设置成人工挖孔。”
- “连续梁主墩使用爬模施工。”

处理规则：

1. 前端将当前场景和用户指令提交给后端自然语言解析接口。
2. 后端解析目标构件范围、构件类型、目标工艺。
3. 匹配成功后更新构件 `method_id`。
4. 若目标工艺不存在，后端可按规则补充工艺模板和资源池。
5. 返回变更摘要，包括动作名称、目标工艺、匹配构件数量、前 20 个目标构件名称和告警。

AI 快捷设置失败或部分匹配时，应保留已成功修改内容，并展示未匹配原因。

### 3.7 参数来源摘要与告警

页面应展示结构物参数来源摘要，包括：

- 桥梁名称。
- 参数来源类型和来源标识。
- 桥幅数量。
- 墩台数量。
- 下部结构构件数量。
- 上部结构项数量。
- 跨径表达式。
- 数据质量检查结果和告警。

对于实际工程 API，来源摘要应来自接口元数据；对于补充 Excel 导入，来源摘要应包含文件名、解析时间、目标桥名和数据质量检查。

### 3.8 排程输入联动

项目参数页中的以下变更会影响排程输入：

- 方案名称。
- 项目名称。
- 计划开始日期。
- 求解时限。
- 构件工艺选择。
- 构件工效分组。
- AI 快捷设置产生的构件工艺变更。

发生上述变更后，应清空旧的任务生成结果、求解结果和方案对比结果。后续点击求解时，系统重新生成任务、逻辑关系和资源候选。

## 4. 数据来源与规则

### 4.1 权威数据源

实际工程以项目结构物参数 API 为结构物主数据权威来源。页面初始化时调用该 API 获取项目、桥梁、工区、墩台、构件、尺寸、来源追踪等数据，再映射为排程场景模型。

Excel 导入在实际工程中应作为补充导入、历史兼容或数据校核能力，不作为主数据源。

### 4.2 目标数据模型

结构物参数 API 返回后，应统一映射为以下层级：

```text
ProjectModel
  -> ProjectBridge
    -> WorkSection
      -> StructureModel
        -> ComponentModel
      -> UpperStructureComponent
```

核心字段要求：

| 模型 | 必填字段 | 说明 |
| --- | --- | --- |
| `ProjectModel` | `project_id`, `project_name`, `start_date`, `bridges` | 项目级排程上下文 |
| `ProjectBridge` | `id`, `name`, `order`, `workpoint_type`, `work_sections` | 桥梁或工点 |
| `WorkSection` | `id`, `name`, `order`, `side`, `structures` | 工区或幅别 |
| `StructureModel` | `id`, `name`, `structure_type`, `order`, `components` | 墩台等结构位置 |
| `ComponentModel` | `id`, `name`, `component_type`, `quantity`, `enabled`, `properties` | 可排程构件 |
| `UpperStructureComponent` | `id`, `name`, `structure_type`, `side`, `span_index`, `support_range` | 上部结构展示和辅助规则 |

### 4.3 枚举规则

工点类型：

- `bridge`：桥梁。
- `road`：路基。
- `tunnel`：隧道。

幅别：

- `left`：左幅。
- `right`：右幅。
- `none`：无幅别或整幅。

结构类型：

- `pier`：桥墩。
- `abutment`：桥台。

构件类型：

- `pile`：桩基。
- `cap`：承台。
- `spread_foundation`：扩大基础。
- `ground_tie_beam`：地系梁。
- `middle_tie_beam`：中系梁。
- `pier_body`：墩柱。
- `cap_beam`：盖梁。
- `abutment_body`：桥台。

### 4.4 结构尺寸展示规则

尺寸展示优先级：

1. 读取 `properties.dimensions_m` 中的规范化尺寸。
2. 桩基优先展示直径和桩长。
3. 墩柱优先展示截面/直径、墩高、根数。
4. 普通构件展示长、宽、高、厚、总长等字段。
5. 若没有规范化字段，展示 `properties.raw` 中的原始摘要。
6. 若无有效尺寸，展示 `-`。

所有进入排程计算的长度类数据应统一为米，数量类数据应统一为数值。

### 4.5 任务生成规则

生成排程输入时：

1. 遍历所有桥梁、工区、墩台、下部结构构件。
2. 跳过 `enabled = false` 的构件。
3. 对每个启用构件匹配工艺和工效。
4. 根据工效的工程量来源计算任务工程量。
5. 根据工期算法计算 `duration_days`，最小为 1 天。
6. 使用工艺的 `resource_type` 作为任务候选资源类型。
7. 上部结构不生成下部施工任务。

异常规则：

- 构件没有匹配工艺：生成严重诊断。
- 构件工程量小于等于 0：生成提醒诊断并跳过该构件。
- 资源池中没有可用资源：求解前诊断为不可行。

### 4.6 工艺逻辑联动规则

同一墩台内默认逻辑：

- 承台在同一墩台全部桩基完成后施工。
- 地系梁在同一墩台全部桩基完成后施工。
- 墩柱优先以承台为前置；无承台时回退到地系梁或桩基。
- 中系梁在墩柱完成后施工。
- 盖梁以墩柱、中系梁为前置。
- 桥台台身优先以承台为前置；无承台时回退到桩基；无桩基时回退到扩大基础。

逻辑关系支持：

- `FS`：前置完成后开始。
- `SS`：前置开始后开始。
- `lag_days`：逻辑间隔天数。
- `all`：候选前置全部满足。
- `first_available`：按候选顺序选择第一个可用前置。

## 5. 接口要求

### 5.1 项目结构物参数 API

页面初始化应调用项目结构物参数 API。建议接口：

```http
GET /api/projects/{project_id}/structure-parameters
```

返回建议：

```json
{
  "project": {
    "project_id": "project-001",
    "project_name": "示例项目",
    "start_date": "2026-06-11",
    "bridges": []
  },
  "summary": {
    "source_type": "structure_parameter_api",
    "bridge_count": 1,
    "carriageway_count": 2,
    "support_count": 50,
    "lower_component_count": 329,
    "upper_component_count": 48,
    "span_expression": "9*40+(86+160+86)+12*40"
  },
  "quality_checks": [],
  "warnings": []
}
```

接口返回的 `project` 应可直接映射为页面当前 `ProjectModel`。若实际系统需要返回完整 `ScenarioInput`，则应同时包含工艺库、逻辑库、资源池、里程碑和求解配置。

### 5.2 工艺工效库接口

工艺工效库由后端提供，页面消费标准化后的 `ProcessTemplate[]`。

建议接口：

```http
GET /api/process-library
PUT /api/process-library
```

工艺模板必须包含：

- 构件类型。
- 工艺名称。
- 工艺方法 ID。
- 默认工期算法。
- 默认工程量来源。
- 默认工效值和单位。
- 默认资源类型。
- 工效分组。
- 默认工艺标记。

### 5.3 AI 快捷设置接口

```http
POST /api/apply-process-natural-language
```

请求：

```json
{
  "scenario": {},
  "prompt": "左幅的3#墩和4#墩的桩基工艺设置成人工挖孔"
}
```

响应：

```json
{
  "scenario": {},
  "changes": [
    {
      "action": "指定墩桩基工艺",
      "process_id": "manual_pile",
      "process_name": "人工挖孔",
      "matched_count": 4,
      "targets": ["3#墩-1#桩基"],
      "message": "指定墩桩基工艺：已将 4 个构件设置为“人工挖孔”。"
    }
  ],
  "warnings": []
}
```

### 5.4 Excel 补充导入接口

Excel 导入保留为补充能力：

```http
POST /api/import-bridge-params
```

实际工程中该入口应标识为“补充导入/校核”，不得覆盖权威结构物参数，除非用户具备权限并明确确认。

## 6. 页面状态与交互要求

加载状态：

- 首次进入页面展示加载状态。
- 结构物参数 API 失败时展示错误信息，并允许重试。

修改状态：

- 修改项目参数、工艺、工效后，当前页面即时更新。
- 修改后应标记当前方案已变化，并清空旧求解结果。

告警状态：

- 数据质量告警在参数摘要区域展示。
- 生成任务诊断在模拟结果页展示。
- 严重诊断应阻止或标记求解不可用。

权限建议：

- 普通查看用户只读。
- 计划工程师可修改方案信息、工艺和工效。
- 系统管理员可维护工艺工效库和资源库。

## 7. 验收标准

功能验收：

- 能从结构物参数 API 加载项目结构树，并正确展示桥梁、幅别、墩台、构件、尺寸、工艺、工效。
- 表格筛选能准确定位目标结构物和构件。
- 用户切换构件工艺后，后续生成任务使用新工艺及对应资源类型。
- 用户切换工效分组后，后续生成任务使用新工程量口径和工期计算规则。
- AI 快捷设置能批量修改目标构件，并返回变更数量和未匹配告警。
- 缺少工艺、工程量为 0、结构参数不完整等情况能形成明确诊断，不静默进入求解。

数据验收：

- 当前 demo 样例“渠溪河特大桥”可作为参考：左右幅各 25 个墩台、各 24 跨，上部结构 48 项，下部构件 329 项，并可生成 329 个排程任务。
- 上部结构不生成下部施工任务。
- 下部结构任务数量应等于可排程且启用的下部构件数量。

联动验收：

- 工艺切换后，任务 `process_name`、`productivity_rule_id`、`compatible_resource_types` 更新。
- 工效切换后，任务 `quantity`、`quantity_label`、`duration_days` 更新。
- 计划开始日期修改后，求解结果日期整体重新计算。
- 参数刷新或修改后，旧求解结果不会继续作为当前结果展示。

## 8. 默认假设

- 实际工程已有结构物参数服务，页面不负责录入完整结构物参数。
- 项目参数页面只允许修改排程相关配置，如工艺、工效、方案信息；结构物几何参数以工程主数据为准。
- 下部结构排程为当前首期范围，上部结构先用于展示和辅助规则判断。
- 工艺工效库、工艺逻辑库、资源配置和里程碑配置仍作为独立模块维护，项目参数页面只消费并引用其结果。
